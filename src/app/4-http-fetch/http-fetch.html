In Angular 22, HttpClient uses the fetch API as its default backend. withFetch() is no longer needed, and the legacy XMLHttpRequest backend becomes an opt-in via withXhr().

Before / after

// Angular ≤ 21
provideHttpClient(withFetch())

// Angular 22 - fetch is the default
provideHttpClient()

// Escape hatch if you still need XHR (e.g. upload progress events)
provideHttpClient(withXhr())

Same GET, two backends

Both buttons call GET /customers/1/orders on our API. Open the DevTools Network tab: the first request shows up with type fetch, the second with type xhr. OrderFetchService injects the default HttpClient; OrderXhrService builds its own in one line:

When you still need XHR: upload progress

The fetch API has no upload-progress events, so reportProgress only works for uploads on the XHR backend. Both buttons POST the same 5 MB payload to /uploads: with XHR the bar fills as bytes leave the browser, with fetch it jumps straight from 0% to 100% when the response arrives.

@if (xhrUploadProgress() !== null) {

{{ xhrUploadProgress() }} %

}
@if (fetchUploadProgress() !== null) {

{{ fetchUploadProgress() }} %

}

When fetch wins: keepalive autosave

keepalive is a fetch-only request option: the request outlives the page. Type a draft below, then reload the page - on pagehide the draft is saved twice, once per backend. The browser kills the in-flight XHR, but the keepalive request still reaches the server. Real-world uses: autosave, analytics, telemetry. 

this.http.post(url, draft, { keepalive: true })

What the server received:

  • fetch + keepalive: @if (fetchDraft(); as d) { "{{ d.text }}" at {{ d.receivedAt }} } @else { nothing }
  • XHR: @if (xhrDraft(); as d) { "{{ d.text }}" at {{ d.receivedAt }} } @else { nothing - the request was killed with the page }
